# Topic Scout Repository Guide

## Purpose

This repository is a reusable Topic Scout tool. It is meant to help agents and human operators
create topic-specific research workspaces for literature scouting, paper triage, corpus
maintenance, dashboard generation, and opportunity analysis.

Use this repository when the goal is to:

- turn a raw research intent into a structured topic contract
- discover papers with OpenAlex and citation-graph expansion
- rank candidate papers with an LLM
- build and maintain an accepted paper corpus
- generate a research dashboard and evidence-backed opportunities
- emit task manifests for sequential, Claw-style, swarm, GitHub Copilot, Copilot CLI, or
  Microsoft scouting-style execution

## Command Surface

Primary commands:

1. `make init`
2. `make scout`
3. `make review`
4. `make corpus`
5. `make opportunities`
6. `make dashboard`
7. `python3 scripts/orchestrate.py emit --mode claw`
8. `python3 scripts/orchestrate.py emit --mode swarm`
9. `python3 scripts/orchestrate.py emit --mode copilot`
10. `python3 scripts/orchestrate.py emit --mode copilot-cli`
11. `python3 scripts/orchestrate.py emit --mode microsoft-scouting`

## How Agents Should Implement the Workflow

This repository has two layers of instructions:

1. `AGENTS.md` is the permanent repo-level contract. It explains the reusable tool surface.
2. `TOPIC_AGENTS.md`, `topic.json`, `agents/`, and topic skills are generated by `make init`.
   They define the active research topic and must be read before topic-specific work.

Agents should treat this repo as a workspace generator first and a research workspace second.
If `topic.json` is missing, the first implementation step is always to initialize or ask for the
research intent needed to initialize. If `topic.json` exists, do not overwrite it unless the user
explicitly asks to reset or create a new topic.

### Universal Agent Workflow

Use this workflow for Codex, GitHub Copilot, Copilot CLI, Microsoft scouting-style agents, Claw,
swarm workers, or human operators:

1. Inspect the current workspace state.
   - Check whether `topic.json` exists.
   - If it exists, read `topic.json` and `TOPIC_AGENTS.md`.
   - If generated role briefs exist, read only the role brief relevant to the current task.
   - Check `data/papers.json` and `data/candidates.json` before changing corpus state.

2. Convert user intent into a research contract.
   - Run `make init` for interactive setup.
   - For non-interactive setup, run `python3 scripts/init_topic.py --intent "<intent>"` or pass
     explicit `--topic`, `--goal`, `--audience`, `--include`, `--exclude`, `--years`, and
     `--taxonomy` arguments.
   - Use `--offline` only when no model-backed intent refinement is available.
   - Confirm that generated `topic.json` contains the topic, goal, audience, inclusion rules,
     exclusion rules, publication years, taxonomy, dashboard sections, search queries, and
     scouting strategy.

3. Discover and rank candidate papers.
   - Run `make scout` or `python3 scripts/scout.py`.
   - Use `QUERY="..." make scout` for a targeted supplemental query.
   - Treat `data/candidates.json` as a review queue, not an accepted corpus.
   - Preserve OpenAlex IDs, DOIs, URLs, abstracts, citation counts, discovered-via provenance,
     relevance scores, and cost metadata.

4. Review candidates against the topic contract.
   - Run `make review`.
   - Accept only candidates that satisfy `topic.json` and `TOPIC_AGENTS.md`.
   - Reject adjacent or attractive papers when they violate exclusions or lack required evidence.
   - If approval is required, stop at recommendations unless a human or coordinator approves IDs.
   - Accept approved IDs with `python3 scripts/accept_candidates.py <candidate-id>...`.

5. Maintain the accepted corpus.
   - `data/papers.json` is the accepted corpus of record.
   - Deduplicate by OpenAlex ID, DOI, arXiv ID, title, and source URL where available.
   - Never invent metadata, abstracts, URLs, citations, methods, or conclusions.
   - Keep scout history and cost metadata intact.

6. Rebuild synthesis artifacts only after accepted corpus changes.
   - Run `make corpus` to generate paper notes and `reports/research_report.md`.
   - Run `make opportunities` to generate `data/research_opportunities.json`.
   - Run `make opportunities-check` to validate opportunity structure.
   - Run `make dashboard` to generate `topic-dashboard.html` and `data/dashboard.json`.
   - If no new paper is accepted, do not rewrite reports, opportunities, or dashboards just to
     create activity.

7. Emit task manifests when another runtime will execute the workflow.
   - Use `python3 scripts/orchestrate.py emit --mode claw` for Claw-style role delegation.
   - Use `python3 scripts/orchestrate.py emit --mode swarm` for parallel subagent execution.
   - Use `python3 scripts/orchestrate.py emit --mode copilot` for GitHub Copilot workspaces.
   - Use `python3 scripts/orchestrate.py emit --mode copilot-cli` for Copilot CLI execution.
   - Use `python3 scripts/orchestrate.py emit --mode microsoft-scouting` for Microsoft
     scouting-style research agents.
   - Each manifest includes the topic contract, commands, role brief path, inputs, outputs,
     dependencies, and required contract files.

8. Validate before reporting success.
   - Run the command that corresponds to the changed behavior.
   - Run `make test` after code changes.
   - Verify generated JSON is parseable and structurally useful.
   - Open or inspect generated Markdown/HTML enough to confirm it is substantive, not merely
     present.

### Runtime-Specific Guidance

Codex agents should implement changes directly in this repo, run the relevant commands, and keep
edits scoped. Codex should use `make init`, `make scout`, `make corpus`, `make opportunities`,
`make dashboard`, and `make test` as the primary verification surface. When asked to prepare work
for other agents, Codex should emit the appropriate manifest and inspect it for completeness.

GitHub Copilot agents should start from `data/copilot_tasks.json` when present. If it is missing,
run `python3 scripts/orchestrate.py emit --mode copilot` after `make init`. Copilot should follow
the task dependency order, read the listed `contract_files`, then read the `agent_brief` for the
assigned role. Copilot should modify only the outputs assigned to that task and report any required
human approval before corpus ingestion.

Copilot CLI agents should start from `data/copilot-cli_tasks.json` when present. If it is missing,
run `python3 scripts/orchestrate.py emit --mode copilot-cli` after `make init`. Copilot CLI should
execute one manifest task at a time, run the command listed in the manifest when applicable, and
print concise status with changed files, accepted IDs, rejected IDs, and validation results. It
should not perform dashboard or report regeneration unless the manifest dependency chain has reached
the publish step.

Microsoft scouting-style agents should start from `data/microsoft-scouting_tasks.json` when present.
If it is missing, run `python3 scripts/orchestrate.py emit --mode microsoft-scouting` after
`make init`. These agents should treat the manifest as a research operations plan: query design,
graph discovery, relevance review, library maintenance, synthesis, gap analysis, and publishing are
separate responsibilities. They should preserve evidence provenance and make uncertainty explicit in
review notes and opportunity analysis.

Claw-style agents should start from `data/claw_tasks.json` when present. If it is missing, run
`python3 scripts/orchestrate.py emit --mode claw` after `make init`. A coordinator should assign each
task to the matching role brief under `agents/`. Workers should write only their listed outputs and
should not skip dependency order. The coordinator is responsible for final acceptance and publication
gating.

Swarm agents should start from `data/swarm_tasks.json` when present. If it is missing, run
`python3 scripts/orchestrate.py emit --mode swarm` after `make init`. Swarm workers may operate in
parallel only when manifest dependencies allow it. Query design must complete before discovery;
discovery must complete before review; review must complete before corpus updates; corpus updates
must complete before synthesis, opportunities, and dashboard generation.

Human operators should use the same command surface and review gates. The safest manual path is
`make init`, `make scout`, `make review`, `python3 scripts/accept_candidates.py <ids>`,
`make corpus`, `make opportunities`, `make opportunities-check`, and `make dashboard`.

### Role Responsibilities

`coordinator` enforces `topic.json` and `TOPIC_AGENTS.md`, assigns work, checks dependencies, and
decides whether publication is allowed.

`query_designer` improves `search_queries` without broadening excluded scope. This role should add
specific synonyms, benchmark names, method names, and adversarial boundary queries.

`graph_scout` runs scholarly discovery and citation-neighborhood expansion. This role writes
candidate papers with provenance and does not accept papers into the corpus.

`relevance_reviewer` applies inclusion and exclusion rules to `data/candidates.json`. This role
returns accepted, rejected, and uncertain IDs with reasons grounded in the topic contract.

`librarian` updates `data/papers.json` only from approved candidate IDs. This role deduplicates,
preserves identifiers and URLs, and keeps scout history intact.

`analyst` runs corpus synthesis, classifies papers into the topic taxonomy, writes paper notes, and
updates `reports/research_report.md`.

`gap_analyst` runs opportunity analysis using `skills/analyze-research-gaps/SKILL.md`. This role
must distinguish corpus gaps from field-wide gaps and validate output with
`make opportunities-check`.

`publisher` regenerates `topic-dashboard.html` and `data/dashboard.json` only after accepted corpus
or opportunity changes justify publication.

### Failure Handling

If OpenAlex is unavailable, keep the current corpus unchanged and report the failed discovery step.
If model-backed ranking fails, preserve OpenAlex candidates and either retry with an available
provider or mark the run as heuristic/offline. If opportunity generation fails, do not publish a
dashboard that implies fresh opportunity analysis. If JSON validation fails, fix the generated JSON
or rerun the generator before claiming success.

When topic scope is ambiguous, agents should tighten the contract rather than broaden the corpus.
When evidence is missing, agents should record uncertainty and future scout queries instead of
inventing conclusions.

## Repo Rules

- Treat the repository root as the live generated workspace surface.
- Do not assume the current topic; inspect `topic.json` and `TOPIC_AGENTS.md` after `make init`.
- `AGENTS.md` is permanent repo-level guidance.
- `TOPIC_AGENTS.md` is generated topic-specific guidance.
- Generated role briefs live under `agents/`.
- Generated topic-specific skills live under `skills/topic-paper-scout/` and
  `skills/analyze-research-gaps/`.
- Example outputs live under `examples/` and should not be treated as the active workspace.

## Validation

- Do not stop at implementation alone.
- Run the relevant commands yourself.
- Verify generated outputs are structurally correct and substantively usable before claiming
  success.
